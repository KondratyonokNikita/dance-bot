# Dance-bot MVP — план

## Проблема
Пользователь состоит в нескольких Telegram-каналах с анонсами мероприятий по социальным танцам (кизомба/бачата и т.п.). Интересуют типы: **вечеринка** (social party / милонга) и **open-air**. Сейчас нужно вручную скроллить чаты, чтобы понять, что есть на конкретный день.

## Цель MVP
Автоматически собирать анонсы из подписанных каналов, извлекать структуру (тип, дата, место) и **писать в структурный лог-файл** (по одному JSON-событию на строку) со ссылкой на исходный пост. Telegram-канал-дайджест, Google Calendar и прочие sink'и — после MVP.

### Scope MVP (зафиксировано)
- **Источники:** [@kredo_dance](https://t.me/kredo_dance), `KIZonEVERYone` — 2 канала.
- **Город:** Минск, таймзона `Europe/Minsk` (UTC+3).
- **Язык анонсов:** только русский.
- **Telegram-аккаунт для Telethon:** основной личный аккаунт пользователя.
- **Историческая загрузка при первом запуске:** последние 7 дней по обоим каналам.
- **Один sink:** JSONL-лог `~/Library/Logs/dance-bot/events.jsonl`. По одной строке JSON на событие. Без Telegram-канала и без Google Calendar на этом этапе.
- Sink Router и `sync_log` в БД проектируем сразу с расчётом на множественные sink'и, но реально подключён только лог-sink.

## Где хостится
Локально на машине пользователя (macOS). Запуск через `launchd` user-agent с `KeepAlive=true`. Логи в `~/Library/Logs/dance-bot.log`. Telethon-сессия живёт на той же машине — миграция на VPS только когда/если понадобится 24/7.

## Стек

| Компонент | Выбор | Обоснование |
|-----------|-------|-------------|
| Язык | Python 3.11+ | Лучшая экосистема под все нужные API |
| Telegram | Telethon (user-session) | Бот не видит сообщения в каналах без админки; user-session это решает |
| LLM | `anthropic` SDK, Claude Haiku 4.5 (`claude-haiku-4-5-20251001`) | Дёшево, structured output через tool use |
| Календарь | `google-api-python-client` *(пост-MVP)* | Не входит в текущий scope |
| Хранилище | SQLite (stdlib `sqlite3`) | Один файл, бэкап = `cp` |
| Конфиг | `pydantic-settings` + `.env` | Типизированный конфиг, валидация при старте |
| Логи | `structlog` | Структурные логи окупаются на дебаге |
| Пакеты | `uv` | Быстрый, lock-файл, ставит Python сам |
| Сервис | `launchd` user-agent | Нативно для macOS, без Docker |

### Чего НЕ берём
- **FastAPI/Flask** — нет нужды в HTTP-сервере.
- **Celery/Redis** — overkill для ~200 сообщений/день.
- **Docker** — добавим только при переезде на VPS.
- **SQLAlchemy** — 3-4 таблицы, raw SQL читается лучше.
- **LangChain** — single-shot extraction короче своим кодом.
- **Postgres** — оправдан только при росте нагрузки/аналитике.

### Ключевые принципы

1. **Source Adapters** изолируют форматы источников. Каждый адаптер возвращает один и тот же `RawMessage` — добавление VK/RSS = ещё один файл, ядро не трогаем.
2. **SQLite — единственная точка интеграции.** Все слои читают/пишут через неё, не друг через друга. Можно остановить любой слой, прогнать заново на исторических данных.
3. **Sink Router** решает, что куда отправлять. Например: `confidence >= 0.8 AND type IN (party, openair)` → Calendar + дайджест; `0.6–0.8` → только дайджест с пометкой «проверь»; `< 0.6` → только в `events` для ручного ревью.
4. **`sync_log`** — отдельная таблица «событие X отправлено в sink Y в момент Z с external_id=...». Даёт идемпотентность: при рестарте бот не задублирует событие в календаре.

### Расширяемость

- Новый источник → новый адаптер, остальное не меняется.
- Новый получатель → новый sink + строка в роутере.
- Поменять LLM или переехать на локальную модель → только `extractor.py`.

## Архитектура парсинга (3 слоя)

```
Telethon listener
  → [Слой 1] keyword filter        # регэксп: вечеринк|пати|party|open[- ]?air|...
  → SQLite (raw_messages)
  → [Слой 2] LLM extractor         # Claude Haiku, structured output
  → [Слой 3] dedup                 # (date, normalized_title, city) или embeddings cos>0.85
  → Google Calendar sync + дайджест-чат
```

SQLite между шагами **обязательна** — позволяет переигрывать парсинг на исторических сообщениях при тюнинге промпта.

### Схема извлечения (Слой 2)

```json
{
  "is_event": true,
  "type": "party | openair | other",
  "title": "...",
  "date_start": "ISO8601 с TZ",
  "date_end": "ISO8601 с TZ | null",
  "location": "...",
  "city": "...",
  "price": "... | null",
  "lineup": ["..."],
  "source_url": "https://t.me/channel/123",
  "confidence": 0.0
}
```

`is_event=false` или `confidence < 0.6` → в очередь на ручной просмотр, **не** в календарь.

## Подводные камни (учесть на старте)

| Грабли | Решение |
|--------|---------|
| «Суббота» без года | Резолвить относительно даты публикации, не «сегодня» |
| Дата в картинке/афише | OCR (Tesseract) или мультимодальный вызов Claude |
| Time zone | Хранить UTC + явный TZ канала |
| Open-air ночь («25 → 26 июня») | Single event с `date_end`, не два |
| «Каждую пятницу» | Не разворачивать в recurring без явного указания |
| Редактирование поста | Telethon ловит `MessageEdited` → обновить событие |
| Отмена/перенос | Детектить ключевые слова, помечать событие |
| Две копии бота с одной сессией | НЕ запускать одновременно — Telegram разлогинит обе |

## Keyword-фильтр (Слой 1) — под социальные танцы

Не техно/рейв. Примерный набор (тюнить на реальных данных):
- Типы событий: `вечеринк|пати|социал(?:ка)?|milonga|милонг|open[- ]?air|опен[- ]?эйр|вечер|фестивал`
- Танцы (контекст): `кизомб|бачат|сальс|зук|урбан|tarrax|semba`
- Время/место маркеры: дата в любом формате, `мск|спб|клуб|студи|школа`

Сообщение проходит дальше, если совпало **хотя бы одно** из «типов» + есть какой-нибудь маркер даты.

## Этапы

1. **Каркас**: `uv init`, конфиг, SQLite-схема, заглушки модулей.
2. **Telegram listener**: авторизация Telethon, чтение @kredo_dance и KIZonEVERYone, запись raw в SQLite.
3. **Слой 1**: keyword filter под социальные танцы, метрика «сколько отсеяли».
4. **Слой 2**: промпт + extractor, прогон на накопленных raw-сообщениях.
5. **Слой 3**: дедуп.
6. **Лог-sink**: запись распарсенных событий в JSONL-файл со ссылкой на оригинал. **Это единственный sink в MVP.**
7. **launchd**: автозапуск, ротация логов, heartbeat-запись раз в сутки.

### После MVP
- Telegram-канал-дайджест (форматированные посты со ссылкой на оригинал).
- Google Calendar sink.
- OCR на афишах (если окажется, что много анонсов в картинках).
- Дополнительные источники / города.

## Открытые вопросы

Все блокирующие закрыты — см. раздел «Scope MVP».

**Можно решать по ходу:**
- Пороги confidence для роутера (стартуем с 0.8/0.6, калибруем на реальных данных).
- Бэкап SQLite (для MVP скорее не нужен — можно пересобрать прогоном истории).
- Локальная LLM как fallback — после MVP, если вообще понадобится.
