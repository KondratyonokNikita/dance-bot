# Dance-bot — текущее состояние проекта

**Дата отчёта:** 2026-06-19

## Что сделано (high-level)

Прошли путь от пустой директории до работающего пайплайна:

```
Telegram (последние 24 ч) → keyword-фильтр → LLM-экстрактор → Google Calendar
```

Пайплайн работает end-to-end на двух каналах: `@kredo_dance` (публичный) и `KIZonEVERYone` (приватный, резолвится через диалоги). Подтверждено на реальных данных — 9 событий из расписания «18-23 июня» корректно извлекаются с типами `party`/`protanzovka`/`openair`/`dance_class`.

Не реализовано (ушло за рамки MVP): persistent-режим работы (бот запускается вручную, не как сервис под `launchd`), SQLite-хранилище raw сообщений (пока всё в памяти), Telegram-канал-дайджест, мониторинг падений / heartbeat.

## Архитектура пайплайна (что реально работает)

```
1. Telethon listener (sync API)
     └─→ итерация по каналам, сообщения за последние 24 часа
2. Keyword filter (filters/telegram.py)
     └─→ regex: вечеринк|протанцовк|open[\s-]?air|party (case-insensitive)
3. LLM extractor (extractor.py)
     └─→ subprocess: claude --print --model claude-haiku-4-5 --output-format json
     └─→ системный промпт читается из prompts/extract_event.md
     └─→ ответ парсится в Extraction { events: list[Event] }
4. Calendar sync (calendar_sync.py)
     └─→ google-api-python-client → events.insert
     └─→ детерминированный event.id для идемпотентности
     └─→ 409 = skip, нет 409 = insert
```

Параллельно с шагом 3 идёт **debug-дамп в stdout** для каждого matched-сообщения: оригинальный текст, сырой ответ LLM, распарсенный JSON. Статусные события (`Logged in`, `Channel scanned`, `Calendar sync`) уходят через `structlog`.

## Схема данных (Event)

```python
class Event(BaseModel):
    event_type: Literal["party", "protanzovka", "openair", "dance_class"]
    dances: list[Literal["bachata", "kizomba", "zouk"]] = []
    date: Optional[str] = None       # YYYY-MM-DD
    time_start: Optional[str] = None # HH:MM (24h, local time)
    time_end: Optional[str] = None   # HH:MM
    location: Optional[str] = None
    price: Optional[str] = None
```

**Принципиальные правила** (закреплены в промпте):
- Извлекаются **все** события с конкретной датой, в прошлом и будущем. Фильтр past/future — на стороне sink-роутера, не в LLM. Это решение пришло после того, как мы поймали баг: «только предстоящие» в промпте давал пустой массив, если канал ошибся с датами.
- Запрещены значения `"other"` и любые выдуманные категории — Pydantic Literal валит на парсинге.
- Мастер-классы / классы / курсы — `dance_class`. Фестивали / концерты / общая реклама — пропускаются.
- Одно физическое событие с несколькими стилями → один `Event` с `dances: ["bachata", "kizomba"]`. Разные мероприятия в одном посте → разные `Event`.

## Промпт-инжиниринг (что меняли и почему)

Ключевые итерации промпта:

1. **Изначально** — единый тип `event_type`, один `dance`. Простая схема для PoC.
2. **+ `protanzovka`** — после того, как стало ясно, что протанцовка ≠ party (неформальная практика, отдельный формат).
3. **`dance` → `dances: list[]`** — один пост мог анонсировать совместное событие «бачата + кизомба».
4. **`time` → `time_start` + `time_end`** — диапазон важен для календаря.
5. **Убрали `"other"`** — модель злоупотребляла им; теперь либо в списке, либо пропуск.
6. **Сняли требование «только предстоящие»** — модель отбрасывала события с уже прошедшей датой, что давало пустой массив на легитимных расписаниях.
7. **Добавили `dance_class`** — раньше курсы пропускались полностью, теперь явный тип.

## Что нужно сделать до первого полного запуска

Бот успешно работает в части Telegram + LLM. Чтобы заработала запись в календарь:

1. **Google Cloud setup** (один раз):
   - Создать проект на [console.cloud.google.com](https://console.cloud.google.com)
   - `APIs & Services` → `Library` → включить **Google Calendar API**
   - `APIs & Services` → `OAuth consent screen` → External, добавить себя в test users
   - `APIs & Services` → `Credentials` → `OAuth client ID` (Desktop app)
   - Скачать JSON и положить как `data/google_credentials.json`
2. **Создать календарь** в [calendar.google.com](https://calendar.google.com) с именем **«Танцы Минск»** (или поменять `google_calendar_name` в `config.py`).
3. **OAuth логин в Google**:
   ```sh
   uv run python scripts/google_login.py
   ```
4. **Запуск бота**:
   ```sh
   uv run dance-bot
   ```

Telegram-сессия уже создана (`data/bot.session`), повторный логин не нужен.

## Известные ограничения / технический долг

- **Квота:** каждый матч → один subprocess-вызов `claude` CLI, идёт через подписку Claude Code. На массовом обстреле может бить по лимитам.
- **Латентность:** subprocess + старт Node на каждое сообщение даёт ~1-3 сек. Для real-time не годится, для часовых батчей — нормально.
- **`claude` в PATH:** при будущем переводе под `launchd` нужно явно прописать PATH, иначе сервис не найдёт бинарь.
- **Структурный лог-sink** (JSONL), который был в плане MVP, **не реализован** — вместо него print-дамп в stdout. Если потребуется, легко добавить отдельный sink в `main.py`.
- **Нет SQLite** — raw-сообщения не сохраняются. Невозможно прогонять промпт «на исторических данных» без повторного запроса в Telegram.
- **Нет дедупа на уровне sink-роутера** — дедуп только в Calendar API через детерминированный event.id. Если завтра добавится TG-дайджест, он получит все сообщения, включая повторы из разных каналов.
- **Запуск только вручную** — `launchd`-сервис не настроен.
